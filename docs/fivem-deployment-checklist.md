# FiveM Deployment Checklist

This checklist is for deploying the core runtime resources into FXServer. It covers the framework loader and compatibility facade only, not server-specific gameplay content.

## Resource Layout

Install these resources under the FXServer resources directory:

```txt
resources/[runtime]/sdb_runtime
resources/[compat]/qb-core
resources/[compat]/qbx_core
resources/[test]/sdb_runtime_smoke
resources/[test]/sdb_qbcore_fixture
resources/[test]/sdb_qbox_fixture
```

The runtime resource must include:

- `fxmanifest.lua`
- `server/main.lua`
- `client/main.lua`
- `web/index.html`
- `web/app.js`
- `config/bootstrap.example.json`

The QBCore facade must include:

- `fxmanifest.lua`
- `server/main.lua`
- `client/main.lua`

The Qbox/QBX facade must include:

- `fxmanifest.lua`
- `server/main.lua`
- `client/main.lua`
- `shared/main.lua`
- `modules/playerdata.lua`

The FXServer smoke resource must include:

- `fxmanifest.lua`
- `server/main.lua`
- `client/main.lua`

The compatibility fixture resources must include:

- `resources/[test]/sdb_qbcore_fixture/fxmanifest.lua`
- `resources/[test]/sdb_qbcore_fixture/server/main.lua`
- `resources/[test]/sdb_qbcore_fixture/client/main.lua`
- `resources/[test]/sdb_qbox_fixture/fxmanifest.lua`
- `resources/[test]/sdb_qbox_fixture/server/main.lua`
- `resources/[test]/sdb_qbox_fixture/client/main.lua`

Verify the checked-in layout before packaging:

```sh
npm run verify:fivem-deployment
```

## Server Config

Core runtime only:

```cfg
ensure sdb_runtime
```

QBCore compatibility mode:

```cfg
ensure sdb_runtime
ensure qb-core
```

Qbox compatibility mode:

```cfg
ensure sdb_runtime
ensure qbx_core
```

Production validation smoke:

```cfg
ensure sdb_runtime
ensure qb-core
ensure qbx_core
ensure sdb_runtime_smoke
ensure sdb_qbcore_fixture
ensure sdb_qbox_fixture
```

The `qb-core` facade declares `provide 'qb-core'` and `dependency 'sdb_runtime'`. The `qbx_core` facade declares `provide 'qbx_core'` and `dependency 'sdb_runtime'`. They are compatibility glue; `sdb_runtime` remains the source of truth.

## Admin Connector Sync

For command-log mode, leave direct command posting disabled and execute emitted commands through the host/RCON integration:

```sh
SDB_FIVEM_SYNC_ENABLED=true
```

The admin connector emits trusted `sdb_runtime_emit ...` envelopes for FXServer console execution.

For direct command execution, configure:

```sh
SDB_FIVEM_SYNC_ENABLED=true
SDB_FIVEM_COMMAND_ENDPOINT=https://fxserver.example.internal/commands
SDB_FIVEM_COMMAND_TOKEN="$COMMAND_TOKEN"
```

Only commands generated by the runtime connector should reach that endpoint.

## Trusted Event Boundary

The runtime resource rejects client-originated control-plane sync events by checking `source ~= 0` before applying them. The verifier checks this for privileged sync events such as:

- `sdb_runtime:syncPermissions`
- `sdb_runtime:syncConfig`
- `sdb_runtime:syncHealth`
- `sdb_runtime:syncMenuTree`
- `sdb_runtime:syncDeployments`
- `sdb_runtime:syncQbPlayerData`
- `sdb_runtime:syncQbShared`
- `sdb_runtime:syncReplicatedState`
- `sdb_runtime:applyAceMirror`

Player-originated events remain limited to menu/session/action requests and are rechecked server-side.

## Boot Checks

After FXServer starts:

1. Confirm `ensure sdb_runtime` loads without manifest errors.
2. Confirm the NUI page opens from `sdb_runtime`.
3. Run `sdb_runtime_health` from the server console.
4. Confirm `sdb_runtime_emit` accepts only JSON envelopes with `sdb_runtime:` event names.
5. Confirm permission, config, menu, deployment, QBCore shared data, and QBCore PlayerData sync events are emitted by the admin connector.
6. In QBCore compatibility mode, confirm a test resource can call `exports['qb-core']:GetCoreObject()`.
7. In Qbox compatibility mode, confirm a test resource can call `exports.qbx_core:GetPlayer(source)` and import `@qbx_core/modules/playerdata.lua`.
8. For production validation, `ensure sdb_runtime_smoke` and run `sdb_runtime_smoke` from the server console; all smoke lines should report `PASS`.

Capture the FXServer console output from the smoke run and verify it locally. The transcript must include FXServer resource lifecycle output for `sdb_runtime`, `qb-core`, `qbx_core`, `sdb_runtime_smoke`, `sdb_qbcore_fixture`, and `sdb_qbox_fixture`; smoke-only fixture files are rejected.

```sh
npm run verify:fxserver-smoke-log -- path/to/fxserver-smoke.log --require-qbcore --require-qbox --require-client
```

For a disposable local Linux smoke run with the pinned FXServer artifact:

```sh
SDB_FIVEM_LICENSE_KEY="your-keymaster-key" npm run smoke:fxserver-core
```

The pinned FXServer tarball is cached under `/tmp/spbox-fxserver-artifact-cache` by default; use `SDB_FXSERVER_ARTIFACT_CACHE_DIR` or `--artifact-cache-dir` to choose another cache location.
`SDB_FXSERVER_ARTIFACT_URL` or `--artifact-url` must be non-empty; blank or whitespace-only artifact URLs fail preflight before staging FXServer.
Runs without `--prepare-only` fail preflight before staging FXServer when `SDB_FIVEM_LICENSE_KEY` is missing.
`--prepare-only` still validates generated-evidence inputs such as port, timing values, and client-required connect host before staging files.
Leave a FiveM client connected during the run when using the default client-required verifier.
The runner waits for the full smoke timeout by default when client checks are required; use `SDB_FXSERVER_SMOKE_COLLECT_MS` or `--collect-ms` only when intentionally shortening a server-only run.
If no players are observed during a shortened client-required run, use `SDB_FXSERVER_CLIENT_GRACE_MS` or `--client-grace-ms` to keep the server open for an additional no-client grace window while player probing continues. The runner extends its effective process timeout to cover startup, the collection window, and that grace window.
`SDB_FXSERVER_SMOKE_TIMEOUT_MS` and `SDB_FXSERVER_SMOKE_COLLECT_MS` must be positive finite numbers, and `SDB_FXSERVER_CLIENT_GRACE_MS` must be a non-negative finite number; invalid timing values fail preflight before staging FXServer.
During client-required runs, keep the terminal visible; it repeats the primary `fivem://connect/...` target plus detected alternate FiveM URIs, such as Tailscale, until the run completes.
For a client on another machine, bind the disposable server to a reachable interface with `SDB_FXSERVER_BIND_ADDRESS=0.0.0.0`, set `SDB_FXSERVER_CONNECT_HOST` to this machine's reachable LAN or public IP, and confirm the selected `SDB_FXSERVER_PORT` is open before running the client-required smoke.
`SDB_FXSERVER_BIND_ADDRESS` or `--bind-address` must be non-empty; blank or whitespace-only bind addresses fail preflight before staging FXServer.
`SDB_FXSERVER_CONNECT_HOST` or `--connect-host` must be non-empty; blank or whitespace-only connect hosts fail preflight before staging FXServer.
The selected `SDB_FXSERVER_PORT` or `--port` must be an integer between `1` and `65535`; invalid ports fail preflight before staging FXServer.
Client-required runs fail preflight before staging FXServer when the advertised connect host is still the unresolved `<server-ip>` placeholder.
Use `/tmp/spbox-fxserver-smoke/connect-info.txt` for the generated `host:port` target, `fivem://connect/host:port` URI, alternate FiveM connect URIs, `connect-launcher.html` path, and detected non-loopback IPv4 addresses from the runner host.
Open the generated `connect-launcher.html` on the operator machine to launch FiveM, try alternate detected IPv4 launch links such as Tailscale when the advertised LAN target is not reachable, and use the reachability table to check browser-side `/info.json` and `/players.json` status for each listed connect target.
For the final client-required gate, `connect-info.txt` must show `Client checks required: yes` and a `Client launcher path:`; server-only connect-info is not valid client-required evidence.
The paths recorded in `connect-info.txt` must be non-empty and must match the transcript, probe, and verifier-output paths supplied to `verify:production-readiness`. The generated connect target and `fivem://connect/...` URI must also match the advertised `/info.json` and `/players.json` probe targets plus the socket-probe port, the advertised connect host must not be the unresolved `<server-ip>` placeholder, and recorded alternate FiveM connect URIs must be well-formed when present. When `connect-info.txt` records `Client launcher path:`, production readiness verifies that launcher file has the same smoke run id, `fivem://connect/...` target, and any recorded alternate FiveM connect URIs. The verifier commands recorded in `connect-info.txt` must include all evidence flags, including network and socket probes.
The `Bind address:` recorded in `connect-info.txt` must be non-empty.
All primary and alternate connect ports recorded in `connect-info.txt` must be integers between `1` and `65535`.
The smoke timing fields recorded in `connect-info.txt` must be numeric, with positive timeout, collection, and effective timeout values plus a non-negative client grace window.
Client-required production readiness requires the complete runner evidence bundle: `connect-info.txt`, `fxserver-network-probe.txt`, `fxserver-socket-probe.txt`, `fxserver-player-probe.txt`, and `fxserver-smoke-verification.txt`.
The transcript and all supplied runner artifacts must carry the same UUID-shaped `Smoke run id:` value; production readiness rejects malformed or mixed log, connect-info, probe, or verifier-output evidence from different smoke attempts.
Use `/tmp/spbox-fxserver-smoke/fxserver-smoke-verification.txt` for the saved verifier output from the run. Client-required runs include the server-side verifier result followed by the client-required verifier result, so a no-client failure still preserves server-side QBCore/Qbox proof. Production readiness requires saved verifier output used as passing evidence to include the runner verifier sections and no verifier failure diagnostics.
Use `/tmp/spbox-fxserver-smoke/fxserver-network-probe.txt` to confirm the runner host can reach the disposable FXServer `/info.json` endpoint for local, advertised, and alternate targets when present. The runner requires this local probe by default; use `SDB_FXSERVER_REQUIRE_NETWORK_PROBE=0` or `--no-require-network-probe` only when that endpoint is intentionally unavailable.
Use `/tmp/spbox-fxserver-smoke/fxserver-socket-probe.txt` to confirm the disposable FXServer listened on both TCP and UDP for the selected port during the smoke run. The runner requires this socket evidence by default; use `SDB_FXSERVER_REQUIRE_SOCKET_PROBE=0` or `--no-require-socket-probe` only when socket inspection is intentionally unavailable.
Use `/tmp/spbox-fxserver-smoke/fxserver-player-probe.txt` to confirm the disposable FXServer local, advertised, and alternate `/players.json` endpoints were reachable and to inspect sampled player counts plus the maximum observed player count during the run. Production readiness verifies that alternate probe targets report HTTP 200 when present, that the recorded sample count matches the sampled lines, and that the maximum observed count matches those samples. Client-required readiness treats a supplied player probe with no sampled players as a no-client failure.
If the verifier reports `No connected FiveM client evidence found`, the FXServer transcript contains server smoke output but no client join/connect signal; rerun the smoke and connect the FiveM client before the timeout expires. Client-required evidence must include both the join/connect marker and the client smoke `PASS` lines.
If it reports `A FiveM client connection was seen, but required client smoke checks are still missing`, collect the client F8 console output because the server saw a client but did not receive the `client:sdb_runtime_smoke:loaded` marker or all client fixture results.
